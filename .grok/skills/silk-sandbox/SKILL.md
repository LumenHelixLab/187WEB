# silk-sandbox — Execution Engine

**Suite:** The 187web Ecosystem v2. Parent index:
[`187web-ecosystem`](../187web-ecosystem/SKILL.md) · Siblings:
`widow-weaver` · `neuro-toxin` · `swarm-mind` · `agent-charlotte`

`silk-sandbox` executes code generated by
[`widow-weaver`](../widow-weaver/SKILL.md) or
[`swarm-mind`](../swarm-mind/SKILL.md) inside isolated containers. Code is
wrapped in "silk" — bounded, monitored, and piped back to the active note or
terminal.

Load this skill when generated code must be validated, tested, or demonstrated
— never run unreviewed code on the host without isolation.

## When to use this

- A markdown code block needs to be executed and results appended.
- Refactored code from `Refactor_Venom` needs unit-test verification.
- A prototype from a specialist persona must run in a safe environment.
- stdout/stderr must stream back in real time for debugging.
- Network or filesystem access must be restricted.

## Security model

### 1. MicroVM Isolation

Execution happens inside **Docker**, **gVisor**, or **Firecracker microVM**
sandboxes — never directly on the host.

| Tier | Backend | Use when |
|------|---------|----------|
| **strict** | gVisor / Firecracker | Untrusted or LLM-generated code |
| **standard** | Docker with `--network=none` | Known-safe project scripts |
| **dev** | Local shell | User explicitly requests host execution |

Default: **strict** for all Charlotte-stack generated code.

### 2. Syscall Monitoring

Block unauthorized operations:

- No outbound network sockets unless `network: allow` is explicitly set.
- No reads outside the sandbox working directory and declared volume mounts.
- No privilege escalation (`sudo`, `setuid`, capability adds).
- Kill and alert (**Widow Red**) on boundary violations.

### 3. I/O Multiplexing

Capture stdout and stderr streams. Pipe results back via WebSocket (Claudian)
or terminal output (agent session). Format as **Ash** blockquotes in Obsidian:

```markdown
> [!quote] silk-sandbox output
> ```
> stdout lines...
> ```
```

Errors use **Widow Red**:

```markdown
> [!danger] silk-sandbox — boundary violation
> Blocked: network socket to 203.0.113.1:443
```

## Execution request schema

```yaml
---
sandbox: silk
tier: strict          # strict | standard | dev
runtime: node         # node | python | bash | docker
timeout_sec: 120
network: deny         # deny | allow
mounts:
  - ./src:/workspace/src:ro
env:
  NODE_ENV: test
---
```

## Supported runtimes

| Runtime | Fence tag | Notes |
|---------|-----------|-------|
| Node.js | `javascript`, `typescript` | `npx tsx` for TS |
| Python | `python` | `uv run` or `python -m` |
| Shell | `bash`, `sh` | No `curl`/`wget` in strict tier |
| Docker | `docker` | Pre-built image only |

## Agent session execution (Grok/Cursor)

When Claudian inline buttons are unavailable, execute via shell with
isolation:

```bash
# Standard Docker isolation
docker run --rm --network=none -v "${PWD}:/workspace:ro" -w /workspace node:22-alpine node script.js
```

**Rules for agent execution:**
1. Confirm code with user before running if it touches filesystem or network.
2. Default `--network=none` and read-only mounts unless write is required.
3. Set `timeout` — kill after 120s unless user extends.
4. Capture exit code, stdout, stderr in the response.
5. Never run `rm -rf`, credential reads, or outbound exfiltration patterns.

## Invocation / workflow example

After [`Refactor_Venom`](../widow-weaver/SKILL.md) produces a testable module:

1. **Review:** User or agent confirms code is intended for execution.
2. **Configure:** `tier: strict`, `runtime: node`, `network: deny`,
   `timeout_sec: 60`.
3. **Execute:** Run tests inside Docker sandbox.
4. **Append:** Pipe results beneath the originating code block.
5. **Alert:** On non-zero exit, flag **Widow Red** and route failures to
   [`Code_Explainer`](../widow-weaver/SKILL.md).

**Activation line format:**

```text
[silk-sandbox] tier=strict runtime=node network=deny timeout=60
```
